# Contributing to Ordinary Client / ROTF

Thanks for helping rebuild ROTF! This is a community reconstruction — there's room for
server devs, AS3/Flash people, content authors, and footage-archaeologists alike.

New here? Read the [README](README.md) for the big picture and [ROADMAP.md](ROADMAP.md)
for what's done and what's open.

---

## Ways to contribute

- **Rebuild content** — author lost ROTF dungeons, bosses, items, classes, and runes as
  server XML (see *Authoring content* below).
- **Server (C#)** — gameplay systems, fixes, performance, the .NET 8 port.
- **Client (AS3/Flash)** — rendering, UI, the Ordinary Client re-skin.
- **Recovery pipeline (Python)** — improve how we mine specs from wikis & footage.
- **Web / deploy** — the Next.js launcher and the AWS deployment path.
- **Docs & triage** — clarify setup, file good issues, verify recovered specs.

---

## Dev setup

**Prereqs:** Linux or WSL, the [.NET 8 SDK](https://dotnet.microsoft.com/download),
`redis-server`, Python 3 (for the pipeline), and the standalone
[Adobe Flash Projector](https://www.adobe.com/support/flashplayer/debug_downloads.html)
to actually play.

```bash
# 1. Backend: redis + app server (:8080) + world server (:2050)
./run-local.sh

# 2. Client: builds client/bin-debug/WebMain.swf (fetches the Flex SDK on first run)
cd client && ./build-client.sh          # or build-client.bat on Windows

# 3. Play: open client/bin-debug/WebMain.swf in the Flash Projector
#    Full walkthrough: docs/HOW_TO_TEST.md
```

Grant yourself admin (for `/spawn`, etc.) after registering an account once:

```bash
./make-admin.sh you@email.com           # then re-login in the client
```

## Authoring content

ROTF content is reconstructed from surviving sources, then authored as server XML.

1. **Find the spec.** Check `docs/DUNGEON_SPECS.json` and `docs/video-recovery/*.json`
   for the recovered boss/dungeon/item data (HP, attacks, taunts, loot, maps).
2. **Pick a free type ID.** New objects use IDs in the `0x8000+` range. **Grep every
   `OrdinaryClient_*.xml` (not just one file) for the ID you intend to use** — a collision
   crashes the server on boot, and the server's dup-key check does *not* catch XML ID
   clashes.
3. **Check for name collisions** against the base game's XML and behaviors before adding a
   boss — a duplicate name is unreachable and can crash boot. Enrich the existing entity
   instead of adding a duplicate.
4. **Author** the XML (objects + behaviors) and, if it's a new sprite, import the sheet on
   the client side too — keep server and client object type IDs aligned.
5. **Boot-test** with `./run-local.sh` and confirm it loads (watch the server log for the
   tile/item/object counts and any errors).

When you can, label recovered mechanics as **grounded** (seen in footage/sources) vs.
**designed** (a plausible fill-in) so reviewers know what's verified.

## The recovery pipeline (Python)

The lost content is reverse-engineered from old gameplay videos and wikis. The heavy
intermediates are **not** committed (see *What's not in the repo*); the pipeline
regenerates them:

```bash
# Frames are extracted from the videos listed in scripts/video/queue.txt
scripts/video/01_fetch.sh        # download a video
scripts/video/02_frames.sh       # extract frames  -> docs/video-recovery/frames/<tag>/
scripts/video/03_analyze.py      # frames -> recovered spec JSON (the committed output)
```

The recovered *specs* (`docs/video-recovery/*.json`) are committed; the raw *frames* are
not (53 GB, reproducible).

## Pull requests

- **Branch** off `main`; keep PRs focused (one boss/dungeon/system/fix per PR is ideal).
- **Build before you push** — `./run-local.sh` should boot cleanly with your change.
- **Match the surrounding style** in whichever language you're touching.
- **Say what you verified** — "boots, loads N objects, spawned the boss in-game" beats
  "should work." For recovered content, cite the source frame/wiki you used.
- Don't commit build outputs, video frames, `.bak` files, or local editor/agent state —
  they're covered by `.gitignore`.

## What's NOT in the repo (and why)

| Excluded | Why | How to get it |
|---|---|---|
| `docs/video-recovery/frames/` (~53 GB) | Spent intermediates | Re-run the video pipeline |
| `videos/` | Large, third-party footage | Source URLs in `scripts/video/queue.txt` |
| `**/bin/`, `**/obj/`, `client/bin-debug/`, `client/.buildsdk/` | Build outputs | Build it |
| `*.bak`, `.claude/`, `MORNING_TODO.md` | Local/personal cruft | n/a |

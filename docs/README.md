# docs/ — ROTF reconstruction notes & recovered specs

ROTF's source never survived, so its content is **reverse-engineered** from surviving
wikis, dev blogs, and gameplay footage. This folder holds the recovered specs and the
notes that drive content authoring. (The 53 GB of raw video frames the JSON was derived
from is **not** committed — it's reproducible via the pipeline; see the root
[CONTRIBUTING.md](../CONTRIBUTING.md).)

## Recovered content specs

| File | What |
|---|---|
| `DUNGEON_SPECS.json` / `DUNGEON_SPECS.md` | Per-dungeon recovered data (bosses, HP, attacks, maps, loot). The primary content reference. |
| `DUNGEONS.md` | Human-readable dungeon overview. |
| `build_progress.md` | Running log of what content has been authored into the server. |
| `taunt_verification.json` | Boss taunts cross-checked against source frames (grounded vs. designed). |
| `mechanics_recovery.json` | Non-taunt layers recovered: attack patterns, maps, loot, stats. |
| `video-recovery/*.json` | Per-video frame-analysis output — the raw recovered specs. |

## Pipeline state (regenerated by the recovery tooling)

| File | What |
|---|---|
| `build_ledger.json` | Tracks which recovered entities have been built. |
| `dev_queue.json` | Work queue feeding the content-authoring loop. |
| `dev_bridge_report.json` | Bridges recovery specs → the dev queue. |

> **Grounded vs. designed:** where possible, recovered mechanics are labelled by how
> certain they are — *grounded* (directly observed in footage/sources) vs. *designed*
> (a plausible reconstruction). Keep that distinction when you add or verify specs.
